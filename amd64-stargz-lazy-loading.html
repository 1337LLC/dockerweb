Example of running Docker images lazily on browser using STARGZ format and container2wasm.

What is STARGZ Lazy Loading?

STARGZ (Seekable TAR GZ) is a container image format that enables lazy loading

  • Faster Startup: Begin running containers immediately without waiting for the full multi-gigabyte image to download.
  • Reduced Bandwidth: Only download the layers and files your application actually uses.
  • On-Demand Content: Additional layers are fetched transparently as needed during execution.
  • Browser-Native: Leverages HTTP range requests (supported by all modern browsers) for efficient streaming.

For more information, see stargz-containers for pre-built STARGZ-compatible images.

How to Use Pre-built STARGZ Images

STARGZ-compatible images are available from the stargz-containers organization packages. These images can be:

  1. Converted to WASM: Use container2wasm to convert STARGZ images to WebAssembly format.
  2. Chunked: Split into smaller chunks for progressive browser loading.
  3. Deployed: Serve chunks over HTTP with proper range request headers.
  4. Loaded Lazily: Browser fetches only needed chunks on-demand.

How It Works

  1. Initial Setup: Container metadata and essential layers load first, allowing immediate interaction.
  2. On-Demand Fetching: When a file or library is accessed, the browser fetches only that chunk via HTTP range requests.
  3. Transparent Loading: Additional layers stream in the background without blocking container execution.
  4. Efficient Caching: Downloaded chunks are cached in memory for reuse during the session.

Performance Benefits

  • Instant Start: Container begins running in milliseconds, not seconds.
  • Minimal Initial Bandwidth: Only a few MB needed to start vs. hundreds of MB for full download.
  • Scalability: Hundreds of users can load containers simultaneously without server overload.
  • Bandwidth Efficiency: Unused layers and files are never downloaded.

Networking

Container networking operates via an HTTP(S) proxy running in the browser:

  • http_proxy, https_proxy, HTTP_PROXY, HTTPS_PROXY are configured to http://192.168.127.253:80
  • Proxy relays requests using the browser's Fetch API
  • Subject to browser CORS policies and limitations
  • For unrestricted networking, see container2wasm documentation

Building Your Own STARGZ Images

To convert your own Docker images to STARGZ format:

# 1. Install eStargz tools
# See: https://github.com/containerd/stargz-snapshotter

# 2. Convert an image to eStargz format
# nerdctl run --rm -v /var/run/docker.sock:/var/run/docker.sock \
#   ghcr.io/containerd/stargz-snapshotter:latest \
#   ctr-remote images convert your-image:tag your-image:stargz

# 3. Convert to WASM using container2wasm
# container2wasm --stargz your-image:stargz your-image.wasm

# 4. Split into chunks (optional)
# split -b 100MB your-image.wasm your-image-chunk-

Advanced Usage

Loading from URL

To load a STARGZ image from a custom URL:

// Modify the script call at the bottom to include your image URL
startLazyWasi("terminal-stargz-lazy", 
  "./src/worker.js"+location.search, 
  "https://your-server.com/your-image-wasm", 
  5, // number of chunks
  {
    fetchProgress: function(loaded, total) {
      console.log(`Downloaded: ${loaded}/${total} bytes`);
    }
  });

Custom Chunk Loading

Implement custom chunk loading logic for advanced use cases like:

  • Prioritizing specific chunks based on execution patterns
  • Integrating with CDNs or edge networks
  • Implementing retry logic with exponential backoff
  • Measuring and optimizing performance metrics
